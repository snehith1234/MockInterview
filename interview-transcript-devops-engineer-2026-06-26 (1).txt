**Suppose after a Helm upgrade a pod is running, but the Service endpoint returns 404 (or the Ingress returns 502). CANDIDATE:
In my Kubernetes/Helm deployment work, if after a Helm upgrade the pod is running but the Ingress returns 502, I would first confirm that the pod is actually Ready, not just Running. If endpoints exist, I would check Service port and targetPort, then curl the Service from inside the cluster. If internal Service traffic works, I move to Ingress and check host rules, path rules, rewrite annotations, ingress class, backend service name, backend service port, DNS, TLS, and load balancer health. Based on the cause, I either fix the Helm values and redeploy or rollback the Helm release if production impact is active.

INTERVIEWER:

**Hard follow-up:** Suppose you find that the Service has endpoints, and the Ingress routes to the correct Service/port, but you still get **502**. CANDIDATE:
For example, after a Helm upgrade, suppose customer-api pods are running and Service endpoints exist, but Ingress returns 502.

I would first compare EndpointSlice IPs and ports with the actual pod IPs and container port. Then I would exec into the pod and confirm the app listens on 8080. I would curl localhost:8080/health, then curl the pod IP from a temporary curl pod, then curl the Service DNS. If those pass, I would check Ingress controller logs. If logs show connection refused, I look at port/protocol mismatch. If logs show upstream timed out, I check app response time, NetworkPolicy, and readiness. If logs show SSL handshake failure, I check whether Ingress is using HTTP while the backend expects HTTPS or vice versa.

INTERVIEWER:
Good—those checks (endpoints/ports, pod-local curl, Service DNS, then Ingress controller logs) are exactly the right direction.  

**Now, when you suspect the issue is an HTTP host/path mismatch (Ingress rule) rather than connectivity, what specific things do you verify in order—using concrete examples of commands/logs—to prove whether the request is hitting the intended Ingress rule and being forwarded to the correct backend path?**

CANDIDATE:
Suppose after a Helm upgrade, users call:

https://api.company.com/customer-api/health

and get 404.

I would check:

kubectl get ingress customer-api -n prod -o yaml

If the Ingress path is /customer-api with pathType: Prefix, I test:

curl -vk -H "Host: api.company.com" https://<lb-dns>/customer-api/health
curl -vk -H "Host: api.company.com" https://<lb-dns>/health

Then I send a correlation ID:

curl -vk \
  -H "Host: api.company.com" \
  -H "X-Correlation-ID: test-404-001" \

I check Ingress logs and app logs. If app logs show:

GET /customer-api/health 404

but internal Service test shows:

curl http://customer-api.prod.svc.cluster.local/health
# 200

then the app expects /health, but Ingress is forwarding /customer-api/health. I would fix the rewrite annotation or app context path through Helm values.
